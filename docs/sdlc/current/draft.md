
Контекст проекта: цифровая платформа КазТБУ
Важное правило по названию
Всегда писать «КазТБУ».
Не писать «КазУТБ», кроме случаев, когда речь идет именно о русском названии.
Общее описание проекта
Мы разрабатываем единую цифровую платформу КазТБУ, а не просто набор отдельных страниц или разрозненных модулей.

Итоговая цель проекта — создать современную, масштабируемую, удобную и централизованно управляемую экосистему университета, которая включает:

Публичную часть сайта для абитуриентов, студентов, родителей, партнеров, сотрудников и гостей.
Внутренние пользовательские зоны / личные кабинеты для разных ролей.
Полноценную админ панель для управления контентом, структурой университета, пользователями, заявками, настройками и другими сущностями системы.
Агенты должны воспринимать проект не как “сделать отдельную страницу”, а как развитие цельного продукта, где каждая новая страница, сущность и функция должна логично встраиваться в общую архитектуру платформы КазТБУ.

Каким должен быть итоговый результат
В конце должен получиться продукт уровня полноценной университетской digital-системы, в которой:

публичный сайт визуально современный и презентабельный;
структура информации понятная и логичная;
контент легко редактируется через админ панель;
есть четкое разделение ролей и прав доступа;
сущности системы унифицированы;
страницы и модули выглядят как части одной экосистемы;
можно масштабировать проект без переделки всей архитектуры;
новые разделы, сервисы и кабинеты можно добавлять без хаоса;
проект выглядит как реальный рабочий продукт для КазТБУ, а не как набор временных UI-экранов.
Как агенты должны воспринимать проект
Если задача сформулирована кратко, агенты по умолчанию должны понимать следующее:

проект относится к КазТБУ;
это единая цифровая система университета;

3. Админ панель
Когда в проекте говорится «админ панель», агенты должны понимать это как полноценный back-office центр управления всей платформой КазТБУ.

Это не просто “страница настроек”, а системная административная среда для работы сотрудников и администрации.

Что по умолчанию означает “админ панель”
Базовая идея
Админ панель — это защищенная часть системы, где уполномоченные пользователи управляют:

Как агенты должны думать о новых страницах
Если создается новая страница, агент не должен воспринимать ее как “одиночный экран”.

Где эта страница находится в общей структуре платформы?
Является ли она публичной или внутренней?
Какая сущность за ней стоит?
Нужна ли для нее админская модель управления?

Можно ли повторно использовать этот паттерн для других сущностей?

Принципы проектирования
Все будущие решения по проекту желательно подстраивать под следующие принципы:

1. Системность
Каждая функция должна быть частью общей экосистемы, а не отдельным несвязанным решением.

2. Масштабируемость
Новые разделы и сущности должны добавляться без архитектурного хаоса.

3. Управляемость
Большинство данных и контента должны управляться через админ панель.

4. Ролевой доступ
Нужно предусматривать разные уровни доступа для разных типов пользователей.

5. Повторное использование паттернов
Если есть удачный шаблон списка, формы, карточки, фильтрации или детальной страницы — лучше использовать единый подход по всей системе.

6. Реалистичность
Интерфейсы и сценарии должны быть пригодны для реальной эксплуатации в КазТБУ.

7. Единый визуальный стиль
Публичная часть и административная часть должны ощущаться как один продукт, даже если визуально различаются по назначению.

8. Ориентация на будущий рост
Любую сущность и страницу желательно проектировать так, чтобы потом можно было:

добавить новые поля;
добавить связи;
расширить права доступа;
встроить аналитику;
интегрировать в админ панель;
подключить новые модули.

Нужно развивать внутреннюю административную систему КазТБУ как центр управления всей цифровой платформой.

А значит, агент должен ориентироваться на:

формы;

настройки;

Что должен понимать агент, когда слышит “сделай страницу”
По умолчанию это означает не просто верстку, а часть общей продуктовой системы.

Агент должен учитывать:

контекст страницы в общей структуре;

как страница будет выглядеть в конечной экосистеме КазТБУ.

Все новые страницы, модули и сущности должны проектироваться как части одной масштабируемой системы.
