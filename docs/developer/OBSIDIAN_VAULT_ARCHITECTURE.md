# Obsidian Vault Architecture for Digital Library

## Purpose
Create a durable project memory system that survives across agent sessions without turning the repository into a markdown dump.

## The three memory layers

### Layer 1 ‚Äî repo truth
Concise startup and execution truth lives in the repository:
- `AGENT_START_HERE.md`
- `project-context/*`
- `.github/copilot-instructions.md`
- `.github/prompts/*` and `.github/instructions/*` for the five-stage SDLC workflow
- `docs/developer/AI_SDLC_WORKFLOW.md`
- runtime and integration code/config

### Layer 2 ‚Äî Obsidian knowledge graph
The vault at `/home/admlibrary/knowledge/kazutb-library-vault` stores the long-term connected understanding:
- architecture
- autogenerated backend/frontend/database/CRM/tooling maps from live repo paths
- domain and business boundaries
- decisions
- incidents
- session history
- handoffs

### Layer 3 ‚Äî operational writeback
Agents write back session outcomes with:
- `bash scripts/dev/obsidian-session-start.sh`
- `bash scripts/dev/obsidian-session-close.sh --summary "..." --verification "..." --next "..."`
- `bash scripts/dev/generate-obsidian-code-maps.sh` or `composer dev:memory-refresh-maps`
- optional Git hook installation via `bash scripts/dev/install-obsidian-hooks.sh`
- rough memory capture via `bash scripts/dev/new-sdlc-draft.sh --type context-update "..."` followed by `/remember`

The closeout script now keeps three synchronized outputs:
1. the daily note in `09-daily-notes/`
2. `11-handoffs/latest-agent-handoff.md`
3. a **timestamped per-task node** under `11-handoffs/task-log/`

## Vault structure
- `00-index` ‚Äî root navigation and startup notes
- `01-project-truth` ‚Äî product identity and non-negotiables
- `02-architecture` ‚Äî runtime/system/tooling maps
- `03-domain` ‚Äî library domain model and rules
- `04-crm-auth-integration` ‚Äî CRM/auth boundaries
- `05-data-quality-stewardship` ‚Äî data risks and workflows
- `06-decisions` ‚Äî dated architectural/product decisions
- `07-bugs-and-incidents` ‚Äî recurring issues and lessons learned
- `08-workstreams` ‚Äî current execution threads
- `09-daily-notes` ‚Äî dated session history
- `09-reference-mirrors` ‚Äî lightweight links back to repo truth
- `10-archive` ‚Äî retired or superseded note structures
- `11-handoffs` ‚Äî latest handoff plus timestamped task-log nodes for each meaningful task
- `12-reference` ‚Äî glossary, paths, templates, checklists

## Rules
- repo truth wins if vault and repo disagree
- avoid secrets in notes
- prefer rich, meaning-carrying file names
- update the vault at the end of each meaningful session
- archive stale structures instead of leaving duplicates alive
