# Obsidian Vault Architecture for Digital Library

## Purpose
Create a durable project memory system that survives across agent sessions without turning the repository into a markdown dump.

## The three memory layers

### Layer 1 — repo truth
Concise startup and execution truth lives in the repository:
- `AGENT_START_HERE.md`
- `project-context/*`
- `.github/copilot-instructions.md`
- `.github/prompts/*` and `.github/instructions/*` for the five-stage SDLC workflow
- `docs/developer/AI_SDLC_WORKFLOW.md`
- runtime and integration code/config

### Layer 2 — Obsidian knowledge graph
The vault at `/home/admlibrary/knowledge/kazutb-library-vault` stores the long-term connected understanding:
- architecture
- autogenerated backend/frontend/database/CRM/tooling maps from live repo paths
- domain and business boundaries
- decisions
- incidents
- session history
- handoffs

### Layer 3 — operational writeback
Agents write back session outcomes with:
- `bash scripts/dev/obsidian-session-start.sh`
- `bash scripts/dev/obsidian-session-close.sh --summary "..." --verification "..." --next "..."`
- `bash scripts/dev/generate-obsidian-code-maps.sh` or `composer dev:memory-refresh-maps`
- optional Git hook installation via `bash scripts/dev/install-obsidian-hooks.sh`
- rough memory capture via `bash scripts/dev/new-sdlc-draft.sh --type context-update "..."` followed by `/remember`

The closeout script now keeps three synchronized outputs:
1. the daily note in `09-daily-notes/`
2. `11-handoffs/latest-agent-handoff.md`
3. a **timestamped per-task node** under `11-handoffs/task-log/`

## Vault structure
- `00-index` — root navigation and startup notes
- `01-project-truth` — product identity and non-negotiables
- `02-architecture` — runtime/system/tooling maps
- `03-domain` — library domain model and rules
- `04-crm-auth-integration` — CRM/auth boundaries
- `05-data-quality-stewardship` — data risks and workflows
- `06-decisions` — dated architectural/product decisions
- `07-bugs-and-incidents` — recurring issues and lessons learned
- `08-workstreams` — current execution threads
- `09-daily-notes` — dated session history
- `09-reference-mirrors` — lightweight links back to repo truth
- `10-archive` — retired or superseded note structures
- `11-handoffs` — latest handoff plus timestamped task-log nodes for each meaningful task
- `12-reference` — glossary, paths, templates, checklists

## Rules
- repo truth wins if vault and repo disagree
- avoid secrets in notes
- prefer rich, meaning-carrying file names
- update the vault at the end of each meaningful session
- archive stale structures instead of leaving duplicates alive

## Second-brain quality bar
The vault should behave like a real **second brain** for the project and its operators, not like a pile of loose notes.

Agents should keep it useful by default:
- capture goals, decisions, key numbers, constraints, done/in-progress/next state, and important ideas when they appear
- normalize rough chat notes or `draft.md` context into linked, durable notes rather than leaving meaning trapped in one long dump
- use descriptive filenames such as `investor-meeting-2026-01.md` or `homepage-branding-decision-2026-04-10.md`, not generic names like `note-48.md`
- answer from this connected context whenever relevant, instead of relying on blank-session guesses
- keep repo docs concise and operational while letting Obsidian hold the richer narrative and cross-links
