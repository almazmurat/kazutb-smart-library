# Obsidian Vault Architecture (Bounded Mirror Model)

**Scope**: Internal-only knowledge organization guidance.  
**Authority boundary**: Repository files remain canonical execution truth.

## 1. Role of the vault

The vault is for human understanding and continuity:
- session logs,
- decisions and rationale,
- incident notes,
- architecture/domain synthesis,
- mirror indexes to repo truth.

The vault is **not** a second execution authority.

## 2. Recommended top-level structure

Current vault (`/home/admlibrary/knowledge/kazutb-library-vault`) can be normalized to this model:

- `00-index` — landing, project index, current-next-step
- `01-current-session` — temporary active-session notes
- `02-project-map` — repo maps and major surfaces
- `03-product-context` — product/domain synthesis
- `04-architecture` — architecture and boundary notes
- `05-domain` — library domain contours/rules
- `06-workstreams` — active workstreams and status
- `07-decisions` — ADR-style decisions
- `08-work-logs` — daily/session logs
- `09-reference-mirrors` — mirrored repo context files
- `10-archive` — retired notes

## 3. Required key notes

- `00-index/home.md` — main landing note
- `00-index/project-index.md` — top-level map of product and repos
- `00-index/current-next-step.md` — currently active next action
- `03-product-context/product-overview.md` — synthesized product intent
- `03-product-context/target-state.md` — end-state contours
- `05-domain/domain-map.md` — entities/workflows/boundaries
- `06-workstreams/current-workstreams.md` — current execution threads
- `07-decisions/decision-log.md` — index of decisions
- `04-architecture/automation-map.md` — scripts/prompts/workflow map
- `10-archive/archive-index.md` — archive entry index

## 4. Link and tag conventions

### Linking
- Prefer wiki links to mirrored canonical files:
  - `[[09-reference-mirrors/repo-context/project-context/06-current-focus.md]]`
- In decision notes, add explicit “Canonical source” block with repo path.

### Tags
- `#mirror` for mirrored canonical content
- `#decision` for ADR-type notes
- `#incident` for bug/incident notes
- `#workstream` for ongoing execution notes
- `#session` for daily/session logs

## 5. What the vault may mirror

- `AGENT_START_HERE.md`
- `project-context/*` canonical set (`00..06`, `98`)
- selected `docs/developer/*` workflow/normalization docs
- `README.md` for navigation context

Mirrors should be generated by scripts, not manually edited as source.

## 6. What the vault must never become

- authoritative stage/focus/contract source
- replacement for `project-context` truth files
- independent API contract authority
- place where runtime/business behavior is changed without repo update

## 7. Sync model

1. Export canonical context from repo → vault mirror:
   - `scripts/dev/export-context-to-vault.sh`
2. Rebuild vault mirror index:
   - `scripts/dev/sync-vault-index.sh`
3. For session closeout:
   - keep narrative logs in vault,
   - keep operational truth updates in repo.

## 8. Safety and reversibility

- No secrets in mirrored notes.
- Mirror scripts are append/overwrite in vault only.
- Repo files are never mutated by vault sync scripts.
- If vault data drifts from repo truth, update vault notes and keep repo unchanged.
