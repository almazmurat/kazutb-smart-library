# Auto-Tracking Instructions for KazUTB Smart Library

**Purpose**: Enable automatic memory capture, state tracking, and git synchronization during agent work sessions without manual user intervention.

**Scope**: These instructions apply to ALL agent interactions with this repository. The agent (Copilot) must follow these rules autonomously.

---

## 🎯 Core Automation Rules

### RULE 0: Obsidian-First Session Bootstrap (MANDATORY)
- **When**: At the START of every agent session, before any analysis or edits
- **What**: Run `bash scripts/dev/obsidian-bootstrap.sh` and read:
  - `artifacts/obsidian/memory-fragments/CENTRAL_HUB_MIN.md`
  - `artifacts/obsidian/memory-fragments/WORK_LOG_YYYYMMDD.md` (if present)
- **Why**: Obsidian is the project second brain and primary operational memory source
- **Enforcement**: Agent must not start implementation before bootstrap memory load

### RULE 1: Automatic State Snapshots
- **When**: After every significant code change, test execution, or architectural decision
- **What**: Capture:
  - Changed files (paths, lines, what changed)
  - Why it changed (business reason, bug fix, feature addition)
  - Impact (affected services, tests, migrations)
  - Verification status (tested? passed?)
- **Where**: Create/update entry in `artifacts/obsidian/memory-fragments/WORK_LOG_YYYYMMDD.md`
- **Format**: YAML frontmatter + markdown bullets
- **Frequency**: Minimum once per major task, maximum every 30 minutes if making rapid changes
- **Strict mode**: Capture even "small" facts (micro-decisions, tiny fixes, failed attempts, command outcomes)

### RULE 2: Continuous Git Tracking
- **When**: Before leaving any task incomplete or finishing a work session
- **What**: 
  - `git add .planning/` if it exists (planning artifacts)
  - `git add app/ config/ routes/ database/` (source changes)
  - `git add tests/` (test updates)
  - `git add artifacts/obsidian/` (memory captures)
- **Message Format**: 
  ```
  [component](type): brief description
  
  - Impact: what broke/fixed/changed
  - Reason: why this change
  - Tests: status of tests
  - Memory: linked to session-YYYYMMDD.md
  ```
- **Frequency**: After every atomic work unit (not every line change)

### RULE 3: Automatic Memory Linking
- **When**: Creating NEW information (findings, decisions, gotchas, insights)
- **What**: Add frontmatter linking to:
  - Category (01-project-truth, 02-architecture, 03-domain, etc.)
  - Related memory fragments (cross-references)
  - Git commit that prompted the memory
- **Where**: Write to appropriate category in `artifacts/obsidian/memory-fragments/`
- **Example**:
  ```yaml
  ---
  type: finding | decision | gotcha | insight
  project: [[Digital Library]]
  date: 2026-04-15
  session: session-2026-04-15
  relates_to: [BookDetailReadService, CatalogSearch]
  git_commit: 06d856b
  tags: [auto-tracked, session-capture]
  links:
    - "[[02-architecture]]"
    - "[[03-domain]]"
  ---
  ```

### RULE 4: Session Boundary Automation
- **When**: At the END of each agent task/session
- **What**:
  1. Run: `bash scripts/dev/obsidian-memory-capture.sh`
  2. Create `session-boundary-YYYYMMDDTHHMM.md` with:
     - Tasks completed (with verification status)
     - Files touched (git diff summary)
     - Decisions made
     - Pending work / blockers
     - Memory fragments generated
  3. Commit all unsaved changes
  4. Push to origin/main
  5. Update `CENTRAL_HUB_MIN.md` with new session reference
- **Git message**: `[session-end] Automatic sync and memory capture for YYYY-MM-DD`

### RULE 5: Heartbeat Persistence (No-Change Sessions)
- **When**: Auto-sync runs and source tree is clean
- **What**: Still create Obsidian heartbeat note (`run-YYYYMMDDTHHMM.md`)
- **Why**: Preserve continuity for every interaction, even with no code diffs

## 🔧 Implementation Specifics

### Auto-Track Categories

Track these **automatically without asking**:

| Category | Trigger | File Location |
|----------|---------|---|
| **Code Changes** | Any git diff | `WORK_LOG_YYYYMMDD.md` |
| **Tests** | Test run output | `WORK_LOG_YYYYMMDD.md` (subsection: "Verification") |
| **Decisions** | "I decided...", "We should..." | `artifacts/obsidian/memory-fragments/decision-YYYYMMDDTHHMM.md` |
| **Gotchas** | "Watch out for...", "This trap caused..." | `artifacts/obsidian/memory-fragments/gotcha-YYYYMMDDTHHMM.md` |
| **Dependencies** | "Now depends on...", "Blocks..." | Embedded in decision/gotcha with cross-links |
| **External API Calls** | POST to CRM, GET EBS platform | `artifacts/obsidian/memory-fragments/integration-note-YYYYMMDDTHHMM.md` |
| **Performance** | "This query takes X ms", "Added index..." | `artifacts/obsidian/memory-fragments/perf-note-YYYYMMDDTHHMM.md` |
| **Security** | "Credential stored in...", "Validated against..." | `artifacts/obsidian/memory-fragments/security-note-YYYYMMDDTHHMM.md` |

### Work Log Template

```markdown
---
type: work-log
project: [[Digital Library]]
date: YYYY-MM-DD
session_id: session-YYYYMMDD
tags: [auto-tracked, continuous-log]
---

# Work Log — YYYY-MM-DD

## Task 1: [Title]
- **Status**: in-progress | completed | blocked
- **Changed Files**: 
  - `app/Services/BookDetail.php` (+45 lines) — why?
  - `tests/Feature/BookDetailTest.php` (+10 lines) — why?
- **Verification**: Test result (passed/failed/pending)
- **Impact**: What does this enable/fix?
- **Decision**: Why this approach over alternatives?
- **Blockers**: None | List what's blocking

## Task 2: [...continuing throughout session...]

## Summary
- **Total files touched**: N
- **Total lines added/removed**: +X -Y
- **Tests status**: N passed, M failed
- **Memory generated**: List fragments created
- **Next**: What should happen next?
```

### Automatic Obsidian Sync Integration

```bash
# This runs automatically at session end:
#!/usr/bin/env bash
set -euo pipefail

cd /home/admlibrary/kazutb-smart-library-main

# 1. Generate from chat logs (if any)
bash scripts/dev/obsidian-memory-capture.sh artifacts/obsidian/memory-fragments

# 2. Commit memory changes
git add artifacts/obsidian/memory-fragments/
git commit -m "[obsidian-auto-sync] Memory fragments updated" || true

# 3. Log session boundary
BOUNDARY_FILE="artifacts/obsidian/memory-fragments/session-boundary-$(date -u +'%Y%m%dT%H%M%S').md"
cat > "$BOUNDARY_FILE" <<EOF
---
type: session-boundary
project: [[Digital Library]]
timestamp: $(date -u +'%Y-%m-%dT%H:%M:%SZ')
tags: [session-end, auto-sync]
---

# Session End — $(date -u +'%Y-%m-%d %H:%M UTC')

Auto-synced by continuous tracking system.
See WORK_LOG_$(date -u +'%Y%m%d').md for details.
EOF

git add "$BOUNDARY_FILE"
git commit -m "[session-boundary] $(date -u +'%Y-%m-%d %H:%M UTC')" || true

# 4. Push everything
git push origin main

echo "✓ Session memory synced"
```

## 📋 Checklist: What Agent MUST Do Automatically

- [ ] **Before any task**: Run `bash scripts/dev/obsidian-bootstrap.sh`
- [ ] **After code edit**: Add entry to WORK_LOG_YYYYMMDD.md
- [ ] **After test run**: Log result in WORK_LOG (Verification section)
- [ ] **After decision**: Create decision-YYYYMMDDTHHMM.md
- [ ] **After finding**: Create finding-YYYYMMDDTHHMM.md
- [ ] **Before git commit**: Ensure memory entry exists with commit message reference
- [ ] **Before task end**: Run obsidian memory capture + session boundary + push
- [ ] **In memory docs**: Always include Obsidian-compatible frontmatter
- [ ] **Cross-linking**: Reference related memory fragments by [[name]]
- [ ] **Categorization**: Always assign to one of 7 categories (01-06, 12)
- [ ] **Timestamps**: Use consistent ISO 8601 format in all docs
- [ ] **No-change interactions**: Persist heartbeat note to Obsidian memory

## 🚫 What NOT To Do

- ❌ Do NOT wait for user to ask for memory capture
- ❌ Do NOT skip git commits because "changes are small"
- ❌ Do NOT leave uncommitted work overnight
- ❌ Do NOT create memory doc without frontmatter
- ❌ Do NOT forget to link decisions to decisions (cross-references)
- ❌ Do NOT use vague descriptions ("fixed issue") — always say WHAT and WHY
- ❌ Do NOT leave test failures undocumented

## 🎯 Expected Agent Behavior

### When working on a task:
```
1. User asks: "Fix the book detail search"
2. Agent:
   a. Reads current WORK_LOG_YYYYMMDD.md to understand session context
   b. Makes the fix in app/Services/BookDetail.php
   c. IMMEDIATELY creates memory entry with what/why/impact
   d. Runs tests
   e. Logs test results in WORK_LOG
   f. If decision was made, creates decision-*.md
   g. Commits with message referencing memory fragment
   h. Continues to next task
```

### When session ends:
```
1. Agent identifies: "No more tasks in queue"
2. Agent:
   a. Reviews all WORK_LOG entries for completeness
   b. Runs obsidian-memory-capture.sh
   c. Creates session-boundary-*.md
   d. Commits and pushes everything
   e. Outputs summary of what was done
```
